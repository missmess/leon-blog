版本控制和代码托管

版本控制和代码托管是实现开发的基石。目前公司内部最常用的组合是git + gitlab，而开源或个人首选的组合是git + github。

版本控制需要提前定义好分支规范 + commit message规范。分支规范是指使用的分支模型。比如常用的master/develop/feature/hotfix模型，或者支持多环境的main/test/ontest/pre模型。

commit规范是针对代码提交的规范，包括commit频次、粒度、message的规范控制。

gitlab是开源代码管理工具，如果公司为了安全性可以考虑部署gitlab到自有服务器上，参考gitlab安装帮助。

包管理仓库

最流行最常用的公有仓库是npm。

往往很多公司都希望搭建自己私有的npm仓库。目前很成熟的开源方案是cnpmcore，可以直接部署到自己公司的服务器上。

多包管理工具

公司中型以上的业务，或者一些基础框架，工具库等，都不会只有一个package。所以就存在了多个package之间，相互依赖的情况出现。这时候各个模块之间的依赖管理、调试等都会出现一些挑战。多包管理分为两种方式：multirepo和monorepo。

multirepo

multirepo就是将每个模块都单独放在自己的仓库内。独立管理、开发、测试、发布。最常见的方式就是git+npm，通过npm安装依赖的包，每个包都使用git管理。如果希望调试它的子依赖库的变更，那么可以通过npm link的方式将本地的子依赖库链接到主仓库上。

另一个方案就是git submodule，由git提供。多个库之间仍然是独立管理的。但是通过git submodule将多个子仓库嵌入到主仓库中。可以直接在主仓库中查看各个子模块的代码，可以快速更新子模块。

monorepo

monorepo（单仓库）就是将多个模块都放在一个代码仓库里，使用同一个版本控制工具管理。它可以在一个仓库管理多个库，但是如果仓库越来越大，也可能导致编译性能下降。

常用的一个方案是Lerna，它是一个monorepo的管理工具。可以统一或分别管理多个包的版本和依赖。可以通过依赖提升和共享依赖来减少多包之间相同的依赖安装和更新（减少磁盘空间占用）。

具体需要使用multirepo还是monorepo，需要根据实际情况来选择。一般来看，monorepo适用于那些多包之间依赖关系比较复杂和紧密的项目，比如基础库（li-mesh的node包仓库，非常多的包，且联系紧密，依赖复杂）。对于那些模块比较独立的项目，那可能选择multirepo会更合适。

构建工具

构建工具是前端工程化的核心，它能自动化的处理前端应用开发过程中比较繁琐但又常见的任务。包括但不限于图片优化、代码压缩、css兼容、代码分包、treeshake、打包编译、dev服务、环境变量。

常见的构建工具有：Webpack、Vite。

渐进式开发框架

现在最主流的SPA框架是React和Vue。

当然还有越来越多人使用的SSR应用框架Next和Nuxt。

构建博客或文档站点的SSG应用框架Gatsby、Hugo。

基于你的经验和偏好选择合适的开发框架，它们不会局限你的想象力。

UI组件库

几乎所有的前端应用都需要用到组件库，大型的前端应用几乎都会有自己的组件库。UI组件库就是一系列可以复用和组合的UI组件的合集，它们一起构建出优秀、美观的用户界面。

主流的PC端UI组件库包含ElementUI、Ant-Design、MaterialUI。

其次，移动端UI组件库有Vant、VUX（常用于微信端页面）。

另外，小程序UI组件库有WeUI、Vant Weapp；Uniapp UI组件库有FirstUI、uView。

By the way: css预编译库也是前端必不可少的工具之一，它极大地提升了css开发效率。主流css预编译库有：Sass、Less、UnoCSS、WindiCss、Styled-Components（比较特殊，它是一个css-in-js库）。你可以脱离UI组件库只使用CSS库，但通常都推荐一起使用提升效率。

自建UI组件库，可以clone开源组件库。如Ant-Design代码库，已包含组件源码及基于dumi的组件库网站。我们在开源基础上进行样式重构、新增业务组件等。

项目启动脚手架

选好了构建工具和开发框架，然后就可以从启动脚手架开始你的编码生活（你也可以不使用任何启动脚手架，但是仅限于学习，工作中请不要这么做）。常用的开源启动脚手架：

  • vite-reactts-antd-redux-starter：后台应用的话强烈推荐（是我做的😄），集成了常用的全生态三方库，并且已包含了后台基础页面布局。
  • Vue Cli：官方出品Vue2 + webpack启动脚手架（已经停止维护）。
  • create-vue：新版cli工具，基于Vite，支持Vue2/Vue3。包含的启动模版也很多。
  • CRA（create-react-app）：React官方推荐（but作者支持Ukraine）。支持在社区自己找启动模版。通常搭配craco
  • Umi：企业级应用框架，使用React。
  • Ice：与Umi比较类似，使用React。
  • awesome-vite：一些Vite的各场景启动模版。
  • 或者在github找到不错的模版，直接使用degit工具down下来。

前端团队可以自己搭建自己的启动脚手架，包括cli工具和脚手架模版。cli工具是为了通过命令行的形式配置脚手架模版里的哪些模块放到最终生成的项目中。cli工具推荐直接使用node脚本开发。

模块化和组件化

模块化

模块化开发时，尽量将不同功能拆分成独立的模块和文件。其他模块或文件使用时，通过import（ES6）和require（CommonJs）导入。

组件化

组件化强调的是，使用Vue、React等框架开发时，页面中相对独立的的一部分UI及其对应的逻辑代码，一起作为一个独立组件抽出，以达到代码复用和易维护性的优势。

模块化和组件化开发方式，是前端工程化不可或缺的一部分。

从微前端到微模块

微前端

微前端就是将一个大型应用拆分成一个一个小型的、且可独立运行/调试/发布的前端应用的技术。类似于后端的微服务、Android端的组件化方案。微前端架构中，各个小型应用（称为微应用）是松耦合的，通常会有一个主应用和N个微应用，相互之间通过框架或者中间件交互。

常见的微前端框架有single-spa、qiankun。qiankun比较成熟，api也比较多。

nextjs官方也有自己的微前端形式：Multi Zones，还没有了解过。

微模块

Module Federation带来了一个更细粒度的微前端架构形式，在此之前的微前端架构解决了页面级别的独立和共享。MF方案真正的将这种独立和共享做到了模块级别，就像是引入一个js包。不过在现有项目上使用MF方案需要自己去处理一系列问题，包括：

  1. 需要花些时间处理shared公共依赖；
  2. 远程模块的版本控制是一个问题，它并不像一个npm包你可以选择升级，也明确知道它的改动。有时候远程模块改动了你可能不知道，运行时才抛出一个异常；
  3. 依赖模块貌似没发tree-shake，比如依赖了一个lodash，那就得加载整个lodash。
  4. ...

hel-micro是以sdk的形式支持MF技术的一个新的方案，官方宣传的特性是模块联邦 sdk 化、免构建、热更新、工具链无关。有机会可以试一试。

截止到2023年1月，调研了一下Vite + React上使用MF方案，虽然有vite-plugin-federation插件，但实际测试有无法逾越的问题存在。

mock和API管理

mock

mock用于开发阶段，后端接口还无法提供数据时，或developer想要检查一下异常数据场景时，可以使用一些mock工具，让自己的业务组件代码只需少量或者无修改就能获取到这些假数据。前端常用的mock手段有以下几种：

  1. 在代码中mock：代表性的mock库是mockjs。可以在项目中直接mock各种接口的返回数据，还能拦截XHR和fetch请求，返回你定义的数据。

  2. 浏览器插件：Just Mock是一个chrome插件，可以在访问目标网址时，配置自定义的请求规则，修改response的返回数据。可以一键开启关闭。

  3. 在线mock网站：JSONPlaceholder是一个在线的mock数据网站，可以直接访问预定的接口。缺点是接口比较少，不能自定义。

API管理

API管理工具是开发阶段针对后端接口进行维护管理的工具，目标是为了提升前后端协作效率和减少扯皮。不过现在的API管理工具往往已经包含了mock能力。

Postman是老牌的API管理软件，支持接口管理、分组、环境变量、调试、mock、脚本测试、共享等等。功能很完善。

Apifox是在线的API管理应用，功能和Postman对齐。

Swagger是java可用，优点是从代码注解直接生成。对于java开发者很方便。

BTW：Node.js的Express框架可以用swagger-jsdoc + swagger-ui-express实现上面Swagger的功能。

公司内部也可以创建自己的API管理平台，功能对标Apifox产品。可以附加一些额外的实用功能，比如各接口定义一键导出typescript定义文件。（前端非常重复且耗时的一个工作就是写ts）

低代码

低代码在前端领域的占比很大（而且只会越来越大），并且由于LLM的不断发展，低代码技术将会发生各种各样的新变革。它能很大程度上提升前端人员的开发效率。

详细介绍了一下低代码：低代码现状调研。

比较推荐Lowcode-Engine低代码平台，实际使用后，完全可以满足商用产品和大型应用的部分重复性页面和简单页面，功能比较强大。但是相对的，配置也会繁琐很多。

响应式设计

一个优秀的前端应用必须能支持在不同屏幕下的页面展示体验。

  • 媒体查询

    从css层处理的。在组件中，根据不同的设备屏幕大小来应用不同的css规则，来达到响应式的目标。

  • 路由页面组件适配

    从逻辑层处理的。根据不同的屏幕大小，同一路由地址，使用不同的组件来渲染。适用于移动端页面和PC端页面相差较大的场景。

代码规范和静态检查

  • 代码规范文档：包括代码风格、‌命名规范等，目的是提高代码的可读性和可维护性。

  • 使用ESLint：执行代码静态检查的一个工具，自动检测代码是否符合预设的代码规范。

  • 使用Prettier：自动格式化代码，让团队保持一致的代码风格。

自动化测试

单元测试

单元测试框架用于对组件或模块执行单元测试，确保它们的功能是没有bug的。单元测试可以使用jest，dom相关状态测试使用jest-dom、React组件测试使用@testing-library/react。

E2E测试

Cypress是一个前端E2E（end-to-end，端到端）测试工具。

性能检测和优化

toC的应用几乎无法避开性能这个话题。性能不好的网站将会大大降低用户体验。

开发阶段：常用的性能检测工具有chrome devtools、lighthouse、vue devtools、react profiler等等。

生产阶段：performance API是浏览器内置的性能数据工具。web-vitals是基于performance的out-of-box库，可快速获取FCP、LCP、FID、CLS等常用性能指标并上报。

性能优化

性能优化方案是一个复杂的专题，跨域很多个纬度，大致的内容包含：

  1. 组件懒加载、图片懒加载：

    • 只有在需要时才加载组件或图片，减少初始加载时间。
    • 使用Intersection Observer API实现懒加载。

  2. 超长列表使用虚拟列表技术：

    • 对于超长列表，使用虚拟列表技术（如React Virtualized）只渲染可见区域的元素，减少DOM节点数量。

  3. 合并资源减少HTTP请求次数：

    • 将多个CSS、JavaScript文件合并为一个文件，减少HTTP请求次数。
    • 使用工具如Webpack进行资源打包。

  4. 代码minify和gzip压缩：

    • 使用工具如UglifyJS、Terser对JavaScript代码进行minify，减少文件大小。
    • 在服务器端配置gzip压缩，减少传输文件的大小。

  5. 图像优化：

    • 使用现代图像格式如WebP，减少图像文件大小。
    • 使用图像压缩工具如TinyPNG。

  6. 缓存优化：

    • 常用数据缓存到本地（如LocalStorage、IndexedDB），减少请求消耗。
    • 配置HTTP缓存（Cache-Control和Expires），减少重复请求。

  7. 使用CDN：

    • 将静态资源存储到CDN，利用CDN的分布式节点加速资源加载。

  8. Service Worker：

    • 使用Service Worker实现接口缓存和离线支持，提高应用的可靠性和性能。

  9. 服务器端渲染（SSR）和静态站点生成（SSG）：

    • 使用SSR和SSG技术提升首屏渲染速度，减少白屏时间。

  10. HTTP/2和HTTP/3：

    • 使用HTTP/2或HTTP/3协议，利用多路复用、头部压缩等特性提高传输效率。

  11. 减少重排和重绘：

    • 优化CSS和JavaScript，减少不必要的DOM操作，避免频繁的重排和重绘。

  12. 异步加载和预加载：

    • 使用async和defer属性异步加载JavaScript文件。
    • 使用<link rel="preload">预加载关键资源。

  13. Vue性能优化实践。

  14. React性能优化指导。

通常一个大型项目内部，可以自己制作一个lightweight、用于线上的性能监控工具。监控一系列性能数据，并及时上报到Sentry。

CI/CD

CI/CD即持续集成/持续部署。通过CI/CD工作流，能自动化的将开发人员提交的代码，进行编译、构建、检查、测试等环节，测试通过后自动部署到测试或生产环境中。

通过开源工具Travis-CI、Jenkins能方便实现CI/CD工作流。

现在主流的代码托管仓库也都已经支持了CI/CD工作流。包括github、gitee、gitlab。

生产部署

前端应用最终是需要部署到服务器上，并且提供域名才能访问的。可以通过CI/CD或者手动部署。

传统方式是公司购买自己的真实服务器，再通过内网部署应用到服务器上。目前大部分公司的主流方案是采用云服务器，即购买云服务厂商的服务器资源。再通过工具部署。

对于个人网站或H5应用来说，部署的方案就更多一些。可以自己购买云服务器、购买带宽、安装nginx、安装jenkins、配置域名、配置https等等。

此外，腾讯云提供的云开发服务可以更快速的完成前后端开发到部署上线，不需要买服务器和域名，以及处理一系列复杂的运维操作

如果是静态站点，可以直接购买对象存储服务，将网站托管到对象存储服务里，并配置访问地址。市面上有很多免费的托管平台，比如Netlify、Github Pages。

很多公司会有自己的云资源管理平台。管理私有云、公有云资源。包括云主机、云存储、云数据库、域名服务等等。方便员工在线申请/编辑自己的应用所需要的云资源。

错误监控

错误监控就是收集应用在线上运行时发生的错误和异常。开源的错误监控工具有Sentry。

Sentry分为web界面和sdk。Sentry支持的sdk非常丰富，不仅仅支持前端技术栈。通常一个项目组或一个公司需要一个Sentry服务即可。web端界面可通过docker等部署到服务器上。

Ending

前端工程化是一个复杂、重要、并且不断进化的技术课题，目的就是提升开发效率、提高质量和加强团队合作效率。随着技术的变迁、环境的变化，它包含的内容可能会发生翻天覆地的变化。我们要做的就是与时俱进，不断学习。
